## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | HeroUI `background` / `content1` | App background, TopBar (`bg-content1`) |
| Secondary (30%) | HeroUI `content2` / `content3` | Sidebar, cards, panel backgrounds |
| Accent (10%) | `#17c964` (HeroUI `primary`) | Synced dot, "Connect repo..." action, active environment dot, primary CTA buttons |
| Destructive | HeroUI `danger` (`text-danger`, `bg-danger`) | Error state chip icon only |

Accent reserved for:
- SyncStatusChip "Synced" icon (`text-primary`, `CheckCircleIcon`)
- WorkspaceSwitcher synced status dot (`bg-primary`)
- Active environment dot in environment dropdown (`bg-success` / `bg-primary` ‚Äî already `bg-success`)
- "Connect GitHub" primary CTA button (`color="primary"`)
- "Connect repo..." dropdown item (`text-primary`)

Semantic color map for sync states (all established in Phase 6 ‚Äî do not change):
- `synced` ‚Üí `text-primary` / `bg-primary` (#17c964 green)
- `syncing` ‚Üí HeroUI Spinner (default foreground)
- `conflict` ‚Üí `text-warning` / `bg-warning` (HeroUI yellow)
- `error` ‚Üí `text-danger` / `bg-danger` (HeroUI red)
- `local` ‚Üí `text-default-400` / `bg-default-300` (HeroUI neutral gray)
- `offline` (new, Phase 7) ‚Üí `text-default-400` / `bg-default-400` (HeroUI neutral gray, one step darker than local to distinguish from local-only dot)

Offline color rationale: CONTEXT D-10 specifies gray cloud-off icon. Use `text-default-400` for the icon (matches `NoSyncIcon` convention for non-error neutral states). WorkspaceSwitcher offline dot uses `bg-default-400` (vs `bg-default-300` for local) to distinguish the two gray states at a glance.

Source: `tailwind.config.js`, `SyncStatusChip.tsx`, `WorkspaceSwitcher.tsx`, CONTEXT D-10

## Component Inventory

Components modified in this phase (not created from scratch):

### SyncStatusChip (extend)

File: `src/features/sync/SyncStatusChip.tsx`

New state: `offline`

| Property | Specification |
|----------|--------------|
| Icon | Inline SVG `CloudOffIcon`, 16x16, `className="text-default-400"` |
| Label | `"Offline"` |
| Button variant | `"flat"` (same as all other non-local states) |
| Button size | `"sm"` |
| isDisabled | `false` ‚Äî click triggers sync attempt (CONTEXT D-12) |
| onPress behavior | Calls `triggerSync(workspaceId)` ‚Äî same as other clickable states |

CloudOffIcon SVG spec: A cloud outline with a diagonal slash line through it. Stroke color via `currentColor`. `strokeWidth="1.5"`. `strokeLinecap="round"`. Size 16x16, viewBox "0 0 16 16". Stroke only, no fill.

Click-while-offline behavior: When status is `offline` and user clicks the chip, `handlePress` calls `triggerSync`. If connectivity returned, sync succeeds and status transitions to `syncing` then `synced`. If still offline, status stays `offline`. No special disabled state ‚Äî same chip, same press handler.

### WorkspaceSwitcher statusDot (extend)

File: `src/components/layout/WorkspaceSwitcher.tsx`

Add `offline` to the `colorMap` in `statusDot()`:

```

```

The existing fallback `bg-default-300` continues to handle `local` and any unknown states.

### syncStore (extend)

File: `src/stores/syncStore.ts`

SyncStatus type change: add `'offline'` to the union.

```typescript
export type SyncStatus = 'synced' | 'syncing' | 'conflict' | 'error' | 'local' | 'offline';
```

Transition toast behavior (CONTEXT D-11):
- When status transitions TO `offline` (previous status was not `offline`): show neutral toast (not error) ‚Äî copy specified in Copywriting Contract below
- When status transitions FROM `offline` TO `syncing`: show neutral toast ‚Äî copy specified in Copywriting Contract below
- Toast style: use `toast()` (neutral), not `toast.error()` ‚Äî offline is expected, not a failure

Previous status tracking: the store must track `previousStatus` per workspace to detect transitions. Compare incoming status to last recorded status for that `workspaceId` before updating `syncStatuses`.

### App.tsx or equivalent root (extend)

File: `src/App.tsx`

Add focus listener in a `useEffect` with empty dependency array. Attaches `getCurrentWindow().onFocusChanged()` from `@tauri-apps/api/window`. On `focused === true`, reads active workspace from `useWorkspaceStore.getState()` and calls `triggerPull` if workspace is GitHub-backed (not local, has `clone_url`). Returns unlisten cleanup function.

Pattern confirmed in RESEARCH.md Pattern 3.

## Interaction States

### SyncStatusChip ‚Äî all 6 states

| State | Icon | Label | Clickable | Color |
|-------|------|-------|-----------|-------|
| `local` | NoSyncIcon (dashed circle with dash) | "Local only" | No (isDisabled) | `text-default-400` |
| `syncing` | HeroUI Spinner size="sm" | "Syncing" | Yes | Spinner default |
| `synced` | CheckCircleIcon | "Synced" | Yes | `text-primary` |
| `conflict` | WarningTriangleIcon | "Conflict" | Yes | `text-warning` |
| `error` | XCircleIcon | "Error" | Yes | `text-danger` |
| `offline` | CloudOffIcon | "Offline" | Yes | `text-default-400` |

### Automatic sync ‚Äî invisible interaction

The debounce push is fully invisible. No UI changes during the 3-second timer countdown. The chip transitions `synced` ‚Üí `syncing` ‚Üí `synced` only when the push fires. Users see a brief "Syncing" state during the actual git operation; they never see a countdown or queued indicator.

### Offline queue ‚Äî invisible interaction

When offline, changes continue saving locally. The chip shows `offline`. No queue indicator, no pending-changes badge, no "N changes queued" label. The mental model is "your work is saved" ‚Äî not "your work is in a queue". This matches CONTEXT specifics: "Offline should feel safe, not broken."

### Focus-pull ‚Äî invisible interaction

When the app regains focus, `triggerPull` fires silently. If changes are pulled, the conflict toast from Phase 6 appears if files were updated. No other UI feedback. The chip briefly shows `syncing` during the pull operation then returns to `synced`.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Chip label ‚Äî offline | "Offline" |
| Chip label ‚Äî synced | "Synced" (existing) |
| Chip label ‚Äî syncing | "Syncing" (existing) |
| Chip label ‚Äî conflict | "Conflict" (existing) |
| Chip label ‚Äî error | "Error" (existing) |
| Chip label ‚Äî local | "Local only" (existing) |
| Toast ‚Äî going offline | "You're offline ‚Äî changes will sync when reconnected" |
| Toast ‚Äî back online | "Back online ‚Äî syncing..." |
| Toast ‚Äî conflict (existing) | "N files updated from remote" / "1 file updated from remote: {filename}" (Phase 6, unchanged) |
| Toast ‚Äî auth error (existing) | "Sync failed ‚Äî sign in to GitHub and try again" (Phase 6, unchanged) |

Toast style for offline/online transitions: `toast()` (neutral) ‚Äî not `toast.error()`, not `toast.success()`. Rationale: offline is an expected transient state; treating it as error would cause alarm. Source: CONTEXT D-11 "reassure, don't alarm" principle.

No empty state: This phase adds no new views, panels, or data displays. There is no empty state to define.

No destructive actions: This phase introduces no destructive actions. No confirmation dialogs.

No primary CTA: This phase has no new user-initiated CTAs. The SyncStatusChip click-while-offline behavior reuses the existing `triggerSync` pattern.

Source: CONTEXT D-10, D-11, D-12; RESEARCH offline/online toast code examples
