**src/stores/syncStore.ts** — Zustand store:

Type definition:
```typescript
type SyncStatus = 'synced' | 'syncing' | 'conflict' | 'error' | 'local';

interface SyncStatusPayload {
  workspaceId: string;
  status: SyncStatus;
  message: string | null;
  conflictedFiles: string[];
}

interface SyncState {
  syncStatuses: Record<string, SyncStatus>;
  initListener: () => Promise<() => void>;
  triggerSync: (workspaceId: string) => Promise<void>;
  triggerPull: (workspaceId: string) => Promise<void>;
  getStatus: (workspaceId: string, isLocal: boolean) => SyncStatus;
}
```

Implementation:
- `syncStatuses`: `Record<string, SyncStatus>` — keyed by workspace ID
- `initListener`: calls `listen<SyncStatusPayload>('sync-status-changed', (event) => {...})` from `@tauri-apps/api/event`
  - On event: update `syncStatuses[event.payload.workspaceId]` to `event.payload.status`
  - If `event.payload.status === 'synced'` AND `event.payload.conflictedFiles.length > 0`:
    - Fire sonner toast with conflict message per D-09 copywriting:
      - 1 file: `"1 file updated from remote: ${conflictedFiles[0]}"`
      - N files: `"${count} files updated from remote"`
    - Check if any conflicted file matches the active request: compare each path against `useRequestStore.getState().activeRequestMeta`. If the active request's `collectionSlug + '/' + parentPath.join('/') + '/' + slug + '.json'` matches any conflicted file path, call `useRequestStore.getState().loadFromFile(...)` to reload it. NOTE: if the reload is complex, a simpler approach is to call the Rust `load_request` command and `loadFromFile` with the result. The key behavior per D-10 is that the editor updates silently.
  - If `event.payload.status === 'error'`:
    - Fire sonner toast: if message contains "not_authenticated", show "Sync failed -- sign in to GitHub and try again". Otherwise show "Sync failed -- check your connection and try again". Use em-dash (--), not hyphen.
  - Returns the unlisten function
- `triggerSync(workspaceId)`:
  - Immediately set `syncStatuses[workspaceId] = 'syncing'` (optimistic per UI-SPEC interaction contract)
  - Call `syncWorkspace(workspaceId)` from api/sync.ts
  - Do NOT update status on resolve — the Tauri event listener handles final state
  - On catch: only log to console (event listener handles error status from backend)
- `triggerPull(workspaceId)`:
  - Same optimistic syncing pattern
  - Call `pullWorkspace(workspaceId)`
- `getStatus(workspaceId, isLocal)`:
  - If `isLocal` return `'local'`
  - Return `syncStatuses[workspaceId] ?? 'synced'` (default to synced per Research open question 3)

Export `useSyncStore` and `SyncStatus` type.

**src/stores/workspaceStore.ts** — Pull on workspace switch (D-12):

1. Add import at top: `import { useSyncStore } from './syncStore';`
2. In `switchWorkspace` action, AFTER setting `activeWorkspaceId` and BEFORE reloading collection/environment stores, add pull trigger for GitHub workspaces:
   ```typescript
   switchWorkspace: async (workspaceId: string) => {
     set({ activeWorkspaceId: workspaceId });
     // D-12: Pull latest remote changes on workspace switch (GitHub workspaces only)
     const workspace = get().workspaces.find((w) => w.id === workspaceId);
     if (workspace && !workspace.is_local && workspace.clone_url) {
       // Fire-and-forget — pull happens async, status updates via events
       useSyncStore.getState().triggerPull(workspaceId);
     }
     // Reload all workspace-scoped stores
     await useCollectionStore.getState().loadWorkspace(workspaceId);
     await useEnvironmentStore.getState().loadEnvironments(workspaceId);
     // Clear active request — IDs are workspace-scoped
     useRequestStore.getState().clearActiveRequest();
   },
   ```
   Note: `triggerPull` is fire-and-forget (not awaited). The UI updates via events. The collection/environment reloads happen immediately with current local state. When pull completes, if files changed, the collection store should be refreshed. The sync event listener in syncStore can trigger a collection reload after successful pull if needed — but for Phase 6 manual trigger, the user can click sync again if they see "Synced" but data seems stale. Full auto-refresh on pull is a Phase 7 concern.

   IMPORTANT: The `triggerPull` call must only fire when the `workspaceId` actually changes (not on re-render). Since this is a Zustand action called explicitly by `handleAction` in WorkspaceSwitcher, it only fires on actual user switch. This avoids Pitfall 7 from Research.
