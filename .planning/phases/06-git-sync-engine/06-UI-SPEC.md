## Interaction Contract

### Sync Chip Click (D-07)

1. User clicks chip in any non-`local_only` state.
2. `syncStore.setSyncStatus(workspaceId, 'syncing')` immediately (optimistic) — chip shows spinner + "Syncing" without waiting for Rust.
3. `invoke('sync_workspace', { workspaceId })` sent to Rust.
4. Tauri event `sync-status-changed` emitted by actor with final state → `useSyncStore` updates.
5. If conflict: toast fires (see Copywriting), status resolves to `synced`.
6. If error: toast fires (see Copywriting), status resolves to `error`.

Visual rule: Spinner replaces icon in `syncing` state — do not show icon + spinner together.

### Workspace Switch Pull (D-12)

Triggered automatically in `workspaceStore.switchWorkspace()` — no new UI chrome. Status chip reflects `syncing` → `synced` or `error` during pull. User receives no modal or confirmation.

### Conflict Toast (D-09)

Triggered once per conflict resolution batch, not once per file. Sonner toast, non-blocking, auto-dismiss at 5 seconds. Single-file example: "1 file updated from remote: users-api/get-user.json". Multi-file example: "3 files updated from remote".

Toast does NOT include "Conflict" in language — uses "updated from remote" framing per CONTEXT.md specifics.

### Active Request Reload (D-10)

If the conflicted file matches `requestStore.activeRequestMeta` file path, `requestStore` reloads the request silently after merge. No toast for this — the file content change is the signal.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA (sync chip, all trigger states) | "Sync" — label on chip while idle/error/conflict: "Synced", "Syncing", "Conflict", "Error", "Local only" |
| Synced chip label | "Synced" |
| Syncing chip label | "Syncing" |
| Conflict chip label | "Conflict" |
| Error chip label | "Error" |
| Local only chip label | "Local only" |
| Conflict toast — single file | "1 file updated from remote: {relative/path.json}" |
| Conflict toast — multiple files | "{N} files updated from remote" |
| Error toast — push failed | "Sync failed — check your connection and try again" |
| Error toast — auth failure | "Sync failed — sign in to GitHub and try again" |
| Empty state (no conflicts) | No empty state: chip shows "Synced" when nothing to report |
| Destructive actions | None in this phase |

Rules:
- Never use "conflict", "overwrite", or "overwritten" in user-visible copy. Use "updated from remote".
- Never expose commit SHAs, branch names, or git terminology in toasts or chip labels.
- Error toast always includes a next step ("try again" or "sign in").

Source: CONTEXT.md D-09, specifics section ("updated from remote" not "overwritten").
